OctoLLM Documentation

Welcome to the OctoLLM documentation. OctoLLM is a distributed AI architecture for offensive security and developer tooling, inspired by octopus neurobiology.

What is OctoLLM?

OctoLLM mirrors the octopus's distributed nervous system:

  • Central Brain (Orchestrator): Strategic planning and coordination using frontier LLMs
  • Autonomous Arms (Specialized Modules): Domain-specific execution with local decision-making
  • Reflex Layer: Fast preprocessing for common patterns without LLM involvement
  • Distributed Memory: Global semantic memory + local episodic stores per arm

Architecture Overview

OctoLLM is designed as a distributed AI system with hierarchical processing.

System Layers

  1. Ingress Layer - API Gateway + Reflex preprocessing
  2. Orchestration Layer - The Brain (strategic planning)
  3. Execution Layer - The Arms (specialized modules)
  4. Persistence Layer - Distributed memory stores
  5. Observability Layer - Metrics, logs, tracing

For detailed architecture documentation, see the ref-docs/OctoLLM-Architecture-Implementation.md in the repository.

Core Concept

The biological inspiration behind OctoLLM's distributed architecture.

Octopus Neurobiology

Octopuses have ~500 million neurons:

  • Central brain: 40 million neurons (strategic planning)
  • Arms: 240 million neurons distributed across 8 arms (local autonomy)
  • Reflex arcs: Immediate responses without brain involvement

Mapping to OctoLLM

  • Central Brain → Orchestrator: Uses GPT-4/Claude Opus for strategic decisions
  • Arms → Specialized Modules: Domain-specific processing with local LLMs
  • Reflex → Preprocessing Layer: Fast pattern matching without LLM calls

For more details, see ref-docs/OctoLLM-Concept_Idea.md.

System Design

Technical implementation details and design decisions.

Technology Stack

  • Orchestrator: Python 3.11+, FastAPI, SQLAlchemy 2.0
  • Reflex Layer: Rust (performance-critical)
  • Arms: Mixed (Python for AI-heavy, Rust for security-critical)
  • Persistence: PostgreSQL, Redis, Qdrant/Weaviate
  • Observability: Prometheus, Loki, Jaeger

Deployment

  • Development: Docker Compose
  • Production: Kubernetes with namespace octollm

For complete architecture documentation, see ref-docs/OctoLLM-Architecture-Implementation.md.

Reflex Layer

Fast preprocessing for common patterns without LLM involvement.

Status: Phase 1 Sprint 1.1 COMPLETE (v1.1.0)

Features

  • Pattern matching and caching
  • PII/injection detection
  • Request validation
  • Sub-10ms latency target

For implementation details, see services/reflex/.

Orchestrator

The central brain for strategic planning and coordination.

Status: Phase 1 Sprint 1.2 COMPLETE (v1.2.0)

Features

  • Task submission and retrieval
  • Reflex Layer integration with circuit breaker
  • Async SQLAlchemy with PostgreSQL
  • REST API with 6 endpoints

For implementation details, see services/orchestrator/.

Arms (Specialized Modules)

Domain-specific execution modules with local decision-making.

Status: Phase 1 Sprint 1.3 PLANNED

Planned Arms

  1. Planner Arm: Task decomposition
  2. Tool Executor Arm: External actions in sandboxed environments
  3. Retriever Arm: Knowledge base search
  4. Coder Arm: Code generation, debugging
  5. Judge Arm: Output validation and quality assurance
  6. Safety Guardian Arm: PII detection, content filtering

Implementation begins in Sprint 1.3.

API Reference

REST API endpoints for the Orchestrator service.

Endpoints

  • POST /submit - Submit a new task
  • GET /tasks/{id} - Retrieve task status
  • GET /health - Liveness check
  • GET /ready - Readiness check
  • GET / - Root endpoint
  • GET /debug/stats - Debug statistics

For complete API documentation, see docs/api/openapi/orchestrator.yaml.

Configuration Reference

Environment variables and configuration options.

Orchestrator Configuration

  • ORCHESTRATOR_DATABASE_URL - PostgreSQL connection string
  • ORCHESTRATOR_REFLEX_URL - Reflex Layer service URL
  • ORCHESTRATOR_ENABLE_REFLEX_INTEGRATION - Enable/disable Reflex integration
